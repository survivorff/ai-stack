# 🐛 一个 Bug 的完整 debug 链路

> 程序员花 80% 时间在 debug。**这套流水线让"复杂 bug"从 4-8 小时缩到 30-90 分钟**。

---

## 这个工作流解决什么

详见 [→ 场景:Debug](../scenarios/debugging.md)。

简化重述:**复杂 bug**(竞态 / 跨多文件 / 偶现)的标准 debug 流程,**让 AI 替你跑实验**。

---

## 工具栈

| 阶段 | 工具 | 必备 / 可选 |
|------|------|:----------:|
| **症状 / 根因分析** | [Cursor](../tools/ide/cursor.md) Cmd+L 或 [Claude Code](../tools/coding-agent/claude-code.md) | 必备 |
| **跨多文件追踪** | [Claude Code](../tools/coding-agent/claude-code.md)(海外)/ [Aider](../tools/coding-agent/aider.md)(国内) | 推荐 |
| **复杂推理(偶现 bug)** | Claude Opus 4.7 / GPT-5 Pro / DeepSeek R1.5 | 必备 |
| **本地复现 / 跑实验** | 你的终端 + AI 写脚本 | 必备 |
| **测试补充** | Cursor / Claude Code | 必备 |

---

## 完整流程

```
Step 1  ─ 症状 → 根因 ────► AI 三段式分析
Step 2  ─ 复现 bug ───────► AI 帮你写最小复现脚本
Step 3  ─ 实验缩小范围 ────► binary search + 假设验证
Step 4  ─ 修(三档方案) ──► AI 给方案,你选,AI 实现
Step 5  ─ 加测试 ─────────► 防止再回归
Step 6  ─ 事后分析 ───────► 写 doc 给团队 / 给未来的自己
```

**总耗时**:30 分钟 - 2 小时(取决于 bug 复杂度)。

---

## Step 1:症状 → 根因(强制 AI 想清楚再动手)

**工具**:Cursor Cmd+L 或 Claude Code

**Prompt**:

```
我有个 bug:
- 现象:[复述用户/我看到的]
- 触发场景:[什么时候发生]
- 报错:[贴 stack trace / log]
- 相关代码:[贴 / @ 文件]

按下面节奏分析,**不要急着 fix**:

1. 现象层面:这个错误的字面意思是什么?
2. 直接原因:代码哪一行触发了它?
3. 根本原因:为什么这一行被这样触发?(更深的设计问题)
4. 影响范围:同根因可能影响哪些其他地方?
5. 三种修复方案:
   a) 最小改动(治标)
   b) 中等改动(治根因不动设计)
   c) 重构(治设计问题)
6. 推荐哪一种 + 为什么

我看完后告诉你选哪个。
```

⚠️ **80% 的"修完又冒"** 是因为没分清"直接原因 vs 根本原因"。这步逼 AI 想清楚。

---

## Step 2:复现 bug(让 AI 帮你写最小复现脚本)

很多 bug 卡在"我没法稳定复现"。

```
我怀疑这是个 [竞态 / 数据 / 配置] 类 bug。
现有代码 + 日志:[附]

帮我写一个最小复现脚本:
- 不依赖完整环境(用 mock / 内存 DB)
- 5 分钟内能跑
- 跑 100 次能稳定看到 bug 出现率(% 概率)
- Python / Bash / Node 都行,你选适合的

输出:可直接 `bash repro.sh` 跑。
```

→ 跑这个脚本,**先稳定复现 bug** 再修。

---

## Step 3:实验缩小范围(binary search 假设)

**Prompt**:

```
偶现 bug,我没复现稳。代码 + 日志见下:[附]

设计 3 个实验,从大到小缩小:
- 实验 1:验证是否 X 引起(怎么测,怎么观测结果)
- 实验 2:验证是否 Y 引起
- 实验 3:验证是否 Z 引起

每个实验要可执行,告诉我具体跑什么命令 / 看什么数据。
```

→ 跑实验 → 观察结果 → 给 AI → 它再缩小范围。
**5-10 轮迭代,通常能定位到具体那段代码**。

### 进阶:git bisect

如果 bug 是某次上线后出的:

```
帮我用 git bisect 找出"什么时候开始出 bug 的"。
我会跑 10 次每个 commit 看是否复现。
你按 binary search 给我下一个测试 commit。
```

AI 自动算 binary search 步骤,**比手动 git log 快 3-5 倍**。

---

## Step 4:修(三档方案对比)

确定根因后,让 AI 给 3 档:

```
根因确认:[描述]

给我 3 档修复方案对比:

1. **最小 hotfix**(< 1 小时,治标)
   - 改什么文件
   - 风险
   - 上线后还有什么留着
2. **中等修复**(2-4 小时,治根因)
   - 改什么模块
   - 影响范围
   - 测试覆盖范围
3. **重构**(1-3 天,治设计)
   - 重构什么
   - 风险高 / 收益大
   - 是否值得现在做

每档给具体改动 / 时间估 / 风险。
```

**关键判断**:看你的优先级
- 紧急上线 → hotfix
- 这周有时间 → 中等
- 这季度规划 → 重构

选定后让 AI 实现:

```
选 [档]。开始改:
- 改之前先告诉我影响哪些文件 / 测试
- 一次改一处,改完跑测试再下一处
- 出红立刻停下找我
```

---

## Step 5:加测试(防止再回归)

bug 修了,**必须加回归测试**(防止 6 个月后又冒出来)。

```
刚才我们修的 bug 是 [描述]。

帮我加测试:
1. 一个测试**专门复现刚才的 bug**(确认我们真修了)
2. 1-2 个测试覆盖类似的"近邻"场景(防类似 bug)
3. 测试要快(< 1 秒)+ 独立(不依赖外部状态)

测试框架:[vitest / jest / pytest 等]
测试文件位置:[__tests__/ / tests/ 等]
```

→ AI 写测试 → 跑确认通过 → commit。

---

## Step 6:事后分析(给团队 / 未来的自己)

**这一步最容易被忽略,但回报最高**。

```
我刚修了一个 bug:
- 现象:...
- 根因:...
- 修法:...
- 用了多少时间:...

帮我写一份事后分析(给团队 wiki / docs/):

1. **背景**:bug 是什么 / 影响多少用户 / 多久了
2. **根因分析**:为什么会发生(技术 + 流程层)
3. **怎么发现的**(监控 / 用户反馈 / 自查)
4. **怎么修的**(改了什么 + 测试覆盖)
5. **为什么没早点发现**(测试缺失 / monitoring 盲点 / review 漏)
6. **action items**(下次怎么避免类似)

中性语气,不指责具体人,**focus on 系统**。
< 800 字。
```

→ 提到 docs/ 或 团队 wiki / Confluence。

⚠️ **这份文档**是组织级"防止再发生"的关键。
个人级:**学习自己的"思考盲区"** —— 半年回看一遍特别有用。

## 时间和成本

```
Step 1:症状 → 根因           5-15 分钟
Step 2:复现脚本               5-15 分钟
Step 3:实验缩小              10-30 分钟
Step 4:修                    10-60 分钟(看选哪档)
Step 5:加测试                 10-15 分钟
Step 6:事后分析(可选)        10-15 分钟
————————————————————————————
合计:                          30 分钟 - 2 小时
```

**月费(已有 Cursor + Claude Pro 用户)**:debug 是日常,**不是新增成本**。
**单次回报**:复杂 bug 手做 4-8 小时 → AI 0.5-2 小时,**3-8 倍**。

---

## 这个工作流的边界

⚠️ **什么时候不够**:

1. **业务逻辑 bug**:产品规则 AI 不知道,需要你输入
2. **物理硬件 / 网络层**:AI 给不了硬件级建议
3. **生产环境间歇性**:AI 给假设但不能直接调 prod
4. **完全黑盒第三方**:AI 不一定有具体系统知识

## 真实使用案例

我自己遇到的 race condition(国内交易所撮合):
- 跑 50 次出 1 次,日志看不出
- Step 1-2:Claude Opus + 日志分析,AI 提出 3 个假设(15 分钟)
- Step 3:AI 设计 3 个实验脚本(30 分钟跑)
- Step 4:实验 2 稳定复现 → 确认根因(锁顺序错)
- Step 5:hotfix + 中等修复双线(70 分钟)
- Step 6:事后分析提交团队 wiki

**总 1.5 小时**。
**手做估计 4-8 小时**(主要在"反复试不能复现")。
**节省 60-80% 时间**。

---

## 进阶变种

### 变种 A:线上紧急 bug(完全没时间)

跳过 Step 5-6:
- Step 1-4 用最小 hotfix
- 上线后再回头补测试 + 事后分析

### 变种 B:全新代码库 bug(不熟悉项目)

先跑一遍 [→ 古早项目 2 小时上手](./codebase-onboarding.md) 流程理解项目,**再 debug**。
盲目 debug 不熟项目 → 高概率改坏其他地方。

### 变种 C:多人协作的 bug

- AI 帮整理 bug timeline → 同步给团队
- AI 写"我已尝试过 X / Y / Z" → 减少同事重复劳动
- 用 GitHub Discussions / Slack thread 留 trail

---

## 实施前的 Checklist

debug 前问自己:
- [ ] 我能稳定复现这个 bug 吗?(不能 → 先 Step 2)
- [ ] 我有足够日志 / 上下文 给 AI 吗?
- [ ] 我打算花多少时间(决定 hotfix vs 重构)?
- [ ] 修完我会加测试吗?(否则 6 个月后又来一次)
- [ ] 这种 bug 是不是重复出现?(yes → 写事后分析)
