# 🐛 Debug 复杂 Bug

> 写代码占 20% 时间,debug 占 80%。AI 在 debug 这件事上的提效**比写代码大得多**。

---

## 你为什么读到这里?(痛点)

- **报错信息看不懂**:Java stack trace 50 层 / Rust borrow checker 又拒绝你
- **复现 bug 要花半小时配环境**:本地装不好 / 数据不对 / 状态难重置
- **只治表象**:症状改完,根因还在,过几天又冒出来
- **跨服务难追**:一个请求经过 5 个微服务,日志散在 5 个地方
- **偶现 bug**:跑 100 次出 1 次,定位极慢

---

## 推荐方案

### 🥇 方案 A:让 AI 先做"症状 → 根因 → 修复"三段式分析

**适合**: 任何"看起来不太对但说不清"的 bug
**时间**: 10-30 分钟
**工具**: [Cursor](../tools/ide/cursor.md) Cmd+L 或 [Claude Code](../tools/coding-agent/claude-code.md)

**步骤**:
1. 把报错 + 触发场景 + 相关代码一起喂给 AI
2. **强制它先分析根因,不要急着写 fix**(关键 prompt 见下)
3. AI 给出 2-3 个可能根因 → 你判断哪个对
4. 选定根因后让 AI 给修复 + 测试

### 🥈 方案 B:跨多文件 / 长链路 bug → Claude Code

**适合**: 涉及 5+ 文件的链路问题(API → service → DAO → DB)
**工具**: [Claude Code](../tools/coding-agent/claude-code.md)
**预算**: Claude Pro $20+

```
> 用户报告 POST /api/user 偶尔返回 500,
> 错误日志附在 docs/error-2026-05-26.txt。
> 跑一下 grep,跨 src/ 找到所有可能相关的代码,
> 给我画 mermaid 时序图说明请求流程,
> 然后判断 bug 可能在哪一段。
```

Claude Code 自动跑 grep / 读多文件 / 给整链分析。

### 🥉 方案 C:偶现 bug(竞态 / 并发)→ Opus 4.7 深度推理

**适合**: 跑 N 次出 1 次的偶现 bug
**工具**: Cursor 切 Opus 4.7 / GPT-5 Pro / DeepSeek R1.5
**预算**: 任意

**关键 prompt**:
```
这个 bug 100 次跑 1 次出。代码如下:[贴代码]
日志如下:[贴日志]

强制思考:
1. 列出所有可能的并发 / 时序 / 状态条件
2. 对每个条件,设计一个能稳定复现的最小测试
3. 推荐先验证哪一个

不要给 fix。先帮我能稳定复现。
```

## 关键 Prompt 模板

### Prompt 1:症状 → 根因 → 修复

```
我有个 bug:
- 现象:[复述用户/我看到的现象]
- 触发场景:[什么时候发生]
- 报错:[贴 stack trace / log]
- 相关代码:[贴/@ 文件]

按下面节奏分析,不要急着 fix:

1. 现象层面:这个错误的字面意思是什么?
2. 直接原因:代码层面哪一行触发了它?
3. 根本原因:为什么这一行会被这样触发?(更深的设计问题)
4. 影响范围:同根因可能影响哪些其他地方?
5. 三种修复方案:
   a) 最小改动(治标)
   b) 中等改动(治根因但不动设计)
   c) 重构(治设计问题)
6. 推荐哪一种 + 为什么

我看完后告诉你选哪个。
```

**为什么这样**:**80% 的"修完又冒"的 bug,都是因为没分清"直接原因 vs 根本原因"**。这个模板逼 AI 想清楚。

### Prompt 2:报错信息看不懂

```
这个报错我看不懂,帮我"翻译":
[贴完整报错]

请说明:
1. 这是什么类型的错(语法 / 类型 / 运行时 / 配置 / 网络)
2. 字面意思中文翻译
3. 通常是什么导致的(列出 3 个最常见)
4. 我应该先排查哪一个 + 怎么排查
```

### Prompt 3:跨服务请求追踪

```
我有一个跨多个服务的 bug。请求路径:
[client] → [gateway] → [service-a] → [service-b] → [db]

我有以下日志(各服务的)/ 代码(各服务关键函数):
[文件附上]

帮我:
1. 画 mermaid 时序图,标出每一跳的输入输出
2. 找出哪一跳的输入和上一跳的输出对不上
3. 那一跳里可能的 bug 点
```

### Prompt 4:让 AI 设计"复现 + 缩小"实验

```
偶现 bug,我没复现稳。代码 + 日志见下:[附上]

帮我设计 3 个实验,从大到小缩小问题范围:
- 实验 1:验证是否 X 引起(怎么测,怎么观测结果)
- 实验 2:验证是否 Y 引起
- 实验 3:验证是否 Z 引起

每个实验要可执行,告诉我具体跑什么命令 / 看什么数据。
```

## 真实使用 tips

### 1. 别只给 AI 看报错,要看上下文

❌ 直接贴 stack trace 让 AI 看
✅ 同时给:报错 + 触发场景 + 相关代码 + 你已经试过什么

**信息密度上去 → AI 给的答案质量上去**。

### 2. AI 给的方案 ≠ 你直接采用

AI 给的方案常常**第一个看着对、深入有问题**。
- 一定 review 它的根因分析(看你认不认同)
- 改之前自己想 5 分钟"为什么这样改对"
- 改完跑测试 + 手动验证

### 3. 复现失败的 bug 让 AI 帮你写"复现脚本"

```
> 帮我写一个最小的复现脚本(Python 或 shell),
> 能模拟出"用户连续点击两次提交按钮"的场景,
> 来验证我们怀疑的并发 bug。
```

**先稳定复现,再 debug**。

### 4. 长链路 bug 配合 git bisect

bug 是某次上线后出的?
```
> 帮我用 git bisect 找出"什么时候开始出 bug 的"。
> 我会跑 10 次每个 commit 看是否复现,
> 你按 binary search 给我下一个测试 commit。
```

AI 自动算 binary search 步骤,**比手动 git log 快 3-5 倍**。

### 5. 报错日志巨大就丢给 AI 摘要

报错 1000 行?
```
> 这是 1000 行错误日志。
> 帮我:
> 1. 找出最关键的 5-10 行
> 2. 解释整体故障路径
> 3. 推断是什么级别的问题(配置 / 代码 / 网络 / DB)
```

## 真实案例

某次我自己:微服务里一个 race condition,跑 50 次出 1 次。

- 手动 debug 估计 4-8h
- 用 Claude Code(Opus 4.7)+ 上面的 Prompt 4
- 它设计了 3 个实验,我跑了 30 分钟
- 第二个实验稳定复现 → 根因清晰
- 修完 + 加测试,共 1.5h

**节省 60%+ 时间**。

---

## 这套方案的边界

- **业务逻辑 bug**(不是技术 bug):AI 不知道你产品规则,需要你输入
- **物理硬件层**:AI 给不了硬件级建议(GPU / 内存条 / 网络硬件)
- **生产环境的间歇性问题**:AI 能给假设但不能直接调 prod,需要你执行
- **完全黑盒的第三方系统**:对你接的第三方系统,AI 不一定有知识
