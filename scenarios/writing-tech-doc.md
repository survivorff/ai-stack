# 📝 写技术文档 / API 文档

> 工程师都讨厌写文档,但**好文档是工程能力的另一半**。AI 让"写文档"从苦差变成 30 分钟搞定。

---

## 你为什么读到这里?(痛点)

- **写完代码懒得写文档** → 时间长了忘了 → 下个人又问你
- **不知道详细程度**:写得太详细啰嗦,写得太简没人看懂
- **API 文档结构乱**:每个 endpoint 命名 / 描述 / 示例 / 错误码各说各话
- **给非工程师讲技术写得太技术**:产品经理 / 老板看不懂
- **README 永远是半成品**

---

## 推荐方案

### 🥇 方案 A:从代码生成文档(主力)

**适合**: 已有代码 / 想给老项目补文档
**时间**: 30 分钟一份完整文档
**工具**: [Claude Code](../tools/coding-agent/claude-code.md) / [Cursor](../tools/ide/cursor.md)

让 AI 读代码 → 生成"对应的文档草稿" → 你 polish + 加上你的"为什么"。

### 🥈 方案 B:边写代码边写文档(预防)

**适合**: 新功能开发中
**工具**: [Cursor](../tools/ide/cursor.md) Cmd+L

写完一个函数 → Cmd+L:
```
> 给这个函数写 JSDoc / docstring,
> 包括参数、返回值、可能抛的异常、一个使用示例。
> 风格匹配项目其他函数。
```

**这种"随手写"的文档,被维护的概率比"事后补"的高 10 倍**。

### 🥉 方案 C:面向用户写"对外文档"

**适合**: README / 用户文档 / API 公开文档
**工具**: [Claude](../tools/foundation-models/claude.md)(写作自然度第一)

**步骤**:
1. 喂代码 + 喂"目标读者画像"
2. 让 Claude 写"以读者视角"的文档
3. 你 review + 加你的"独到见解"

## 关键 Prompt 模板

### Prompt 1:从代码生成 API 文档

```
这是我们的 API 接口代码(Express / FastAPI / Spring 都行):
[贴代码或 @ 文件]

帮我写一份对应的 API 文档,格式是:

## POST /api/users

### 描述
一句话说明做什么。

### 请求
| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| ... |

### 响应
{
  // 200
  // 4xx 各错误码
}

### 示例
完整 cURL + 完整响应

### 错误码
| Code | 说明 | 何时发生 |
|------|------|----------|

要求:
- 字段说明要"对调用方有用"(不是"username 字段是用户名"这种废话)
- 错误码列出实际可能的全部
- 示例用真实场景的数据,不要 foo / bar
- 中文写
```

### Prompt 2:写 README

```
这是我的项目代码:[@ codebase / 关键文件]

帮我写 README.md,包括:

1. **一段话定位**:这个项目是干什么的?适合谁用?
2. **核心特性**:3-5 条,具体不空洞
3. **快速开始**:5 分钟跑起来的最小步骤
4. **常用使用场景**:2-3 个具体例子(每个有完整代码)
5. **完整 API / 配置参考**(链到子文档)
6. **FAQ**:可能被问的 5 个问题
7. **License / 致谢**

风格:
- 中文
- 不夸张(不写"the most powerful X" 这种)
- 给第一次见到这个项目的人写
- 带 emoji 但克制(每节最多 1 个)
```

### Prompt 3:面向不同读者写

```
这是一段技术内容:[贴]

帮我写 3 版,分别给:

**版本 A**:给 senior 工程师(直接、技术、密度高)
**版本 B**:给 junior 工程师(多解释概念、加示例)
**版本 C**:给产品经理 / 老板(去技术细节、强调业务影响)

每版长度差不多,但用词 / 信息密度不同。
```

**对"同一个项目要给不同人讲"的场景救命**。

### Prompt 4:文档审稿(找问题)

```
这是我写的文档:[贴]

帮我做 4 维度审稿:

1. **完整性**:有没有应该写但没写的(常见漏掉:错误处理 / 配置项 / 限制 / 升级路径)
2. **准确性**:对照下面的代码,文档有没有过时 / 错的地方
   [贴代码]
3. **可读性**:有没有"作者很清楚但读者看不懂"的段落?哪段?为什么?
4. **风格一致性**:称呼 / 时态 / 代码风格在全文一致吗?

每条找出来 + 给修改建议。
```

### Prompt 5:把代码注释生成模式应用到全文件

```
这是我的代码 [@ file]:

帮我:
1. 在每个函数 / 类前面加 JSDoc / docstring
2. 在复杂的代码块上面加 1-2 行 inline 注释解释
3. 不要给已经显然的代码加注释(避免噪音)
4. 风格匹配项目其他文件已有的注释

直接给修改后的代码 diff。
```

## 真实使用 tips

### 1. 文档结构 > 文档内容

读者不会"读完全部文档"。他们会**扫描标题**找 ta 要的部分。
所以:
- 标题要"用户视角"而不是"代码视角"(写"如何处理认证错误"而不是"AuthError 类")
- 每个 H2 / H3 都让人一眼知道"这里有什么"
- 加目录 / 锚点

### 2. 让 AI 写"为什么"比"怎么做"更值

代码本身已经回答"怎么做"。
**好文档的差异化是回答"为什么"**:
- 为什么这么设计(不是 X 不是 Y)
- 这个 API 假设是什么(用错会怎样)
- 历史上为什么有这个 hack

让 AI 写时**显式 prompt 它"加上 Why"**。

### 3. 永远配示例(可运行的)

用户读文档**最先找的是"能不能复制粘贴跑起来"**。
- API 文档:每个 endpoint 配 cURL
- 函数文档:配最小可运行示例
- 配置文档:配完整 example.config

**示例要真实可跑** —— 让 AI 用 `realistic` 数据而不是 foo/bar。

### 4. README 第一屏是命脉

GitHub 上 90% 用户**只看 README 上半屏**就决定要不要 star / 用。
所以 README 顶部:
- Logo / Banner(可选)
- 一段话定位(15 秒说清是什么)
- 一条最 demo 的"30 秒上手"
- 链到详细文档

让 AI 写 README 时**强调"前 100 行决定一切"**。

### 5. CHANGELOG 让 AI 帮你写

```
> 这是 git log 从 v1.0 到 v1.1 的所有 commit:
> [贴 git log]
>
> 帮我写 CHANGELOG.md,按 keep-a-changelog 格式:
> - Added(新功能)
> - Changed(改动)
> - Deprecated(过期)
> - Removed(移除)
> - Fixed(修复)
> - Security(安全)
>
> 给每条加上对用户影响的简短说明。
```

### 6. 别让 AI 写"过度工程化"的文档

AI 默认倾向写"超完整"的文档(因为训练数据里好的开源项目文档都长)。
**但你项目可能不值得这么长**。
prompt 时**显式说"短而精,不要过度详细"**:
> 长度控制在 < 200 行,**只覆盖用户必需信息**,内部细节链到内部 wiki。

## 真实案例

我给 [content-os](https://github.com/survivorff/content-os) 的 xiaohongshu-editor skill 补文档:

- 用 Prompt 1(从代码生成)+ Prompt 2(写 README)
- 30 分钟出 5 个文档(SKILL.md + 4 个 reference)
- 自己 polish 加"我的真实经验和踩坑"(关键的"为什么")
- 总耗时 1.5h,**手写估计 1 整天**

---

## 这套方案的边界

- **完全独创的概念**:AI 只能基于已有训练数据写,真新概念解释你要写
- **公司私有的 know-how**:AI 不知道你公司内部如何做
- **行业特定术语 / 法规**:某些专业术语 AI 用错你要修
- **"代码就是文档"派**:有些团队拒绝写文档,你说服不了 ta 们
