# 📄 写 PRD / 需求文档

> 把"模糊的想法"变成"工程师能直接实现的文档"。**AI 让 PRD 从"写 3 天"变成"写 3 小时"**。

---

## 你为什么读到这里?(痛点)

- **写起来慢**:一份完整 PRD 手写 2-3 天,改 3 轮才定稿
- **团队对齐难**:PM 写的工程师看不懂,工程师觉得"需求不清"
- **模板不一致**:每个 PM 格式不同,新人不知道该写多详细
- **边界条件想不全**:happy path 写得好,error / edge case 漏一堆
- **写完没人看**:太长 / 太抽象,工程师扫一眼就开始"自己理解"

---

## 推荐方案

### 🥇 方案 A:Kiro Spec 模式(推荐给技术 PM / 技术 founder)

**适合**:你自己也写代码 / 你的 PRD 直接给工程师实现
**时间**:2-4 小时(含 review)
**工具**:[Kiro](../tools/ide/kiro.md) Spec 模式
**预算**:$19/月

**步骤**:
1. Kiro 创建 Spec → 输入一句话需求
2. AI 用 EARS 格式扩成 `requirements.md`(含 acceptance criteria)
3. 你 review + 改
4. AI 生成 `design.md`(数据模型 / API / 时序图)
5. 你 review + 改
6. AI 拆成 `tasks.md`(可执行任务列表)
7. 直接进入实现(见 [→ workflows/spec-to-code](../workflows/spec-to-code.md))

**优势**:PRD → 设计 → 任务 → 实现 **一条线不断**,不需要"写完 PRD 再翻译给工程师"。

### 🥈 方案 B:Claude / DeepSeek 对话式(推荐给非技术 PM)

**适合**:你不写代码,PRD 给工程师 / 设计师看
**时间**:1-3 小时
**工具**:[Claude Sonnet 4.7](../tools/foundation-models/claude.md) / [DeepSeek](../tools/foundation-models/deepseek.md)
**预算**:$20/月(Claude Pro)或 ¥0(DeepSeek)

**步骤**:
1. 跟 AI 对话,描述你想做什么
2. AI 帮你扩成完整 PRD(用下面的 prompt 模板)
3. 你 review + 改
4. 粘到 Notion / 飞书文档 → 分享给团队

### 🥉 方案 C:Notion AI / 飞书 AI 内嵌(推荐给已有工具的团队)

**适合**:团队已经在 Notion / 飞书写文档
**时间**:1-2 小时
**工具**:[Notion AI](../tools/notes/notion-ai.md) / [飞书智能伙伴](../tools/notes/feishu-ai.md)
**预算**:含在团队订阅里

**步骤**:
1. 打开 Notion / 飞书文档
2. 从 PRD 模板开始
3. 每节用 AI 续写 / 扩展
4. 团队直接在同一文档 review

## 关键 Prompt 模板

### Prompt 1:从一句话到完整 PRD

```
我要做一个功能:[一句话描述]

背景:
- 产品:[产品名 + 定位]
- 目标用户:[谁]
- 当前痛点:[用户现在怎么做 + 哪里痛]
- 商业目标:[为什么做这个功能]

帮我写一份完整 PRD,结构:

1. **概述**(一段话,让任何人 30 秒看懂)
2. **目标 / 非目标**
   - 目标(这次做什么,可量化)
   - 非目标(明确不做什么,防范围蔓延)
3. **用户故事**(3-5 个,格式:作为 [角色],我想 [动作],以便 [价值])
4. **功能需求**(按模块拆,每条可验证)
5. **验收标准**(WHEN ... THEN ... SHALL ... 格式,至少 8 条)
   - 必须覆盖:正常流 + 错误流 + 边界条件
6. **非功能需求**(性能 / 安全 / 兼容 / 可访问性)
7. **数据需求**(新增 / 修改的数据字段 + 存储)
8. **依赖 / 风险**(技术 / 业务 / 时间)
9. **里程碑**(MVP → P1 → P2)
10. **开放问题**(还没想清楚的,标出来)

要求:
- 中文
- 每节简洁(用表格 / 列表代替长段落)
- 验收标准必须"工程师看了就能写测试"
- 非目标必须写(这是 PRD 最容易漏的)
- 总长度 < 2000 字(太长没人看)
```

### Prompt 2:让 AI 帮你"挑刺"(写完后)

```
这是我写的 PRD:[贴完整版]

帮我做 5 维度审稿:

1. **完整性**:有没有应该写但没写的?(常见漏:错误处理 / 权限 / 数据迁移 / 回滚)
2. **可验证性**:哪些验收标准"工程师看了不知道怎么测"?
3. **歧义**:哪些描述"两个人可能理解不同"?
4. **范围蔓延**:有没有"看起来小但实际很大"的需求?
5. **工程师视角**:如果我是工程师,我会问什么问题?

每条找出来 + 给修改建议。
```

### Prompt 3:让 AI 模拟"工程师提问"

```
你扮演一个资深后端工程师,刚看完这份 PRD:[贴]

你会问 PM 什么问题?(至少 10 个)

分类:
- 技术可行性(能不能做 / 怎么做)
- 边界条件(如果 X 怎么办)
- 性能(量级 / 并发 / 延迟)
- 安全(权限 / 数据泄漏)
- 依赖(第三方 / 其他团队)
- 时间(这个真的 1 周能做完?)

每个问题标"必须回答" / "最好回答" / "可以后续"。
```

### Prompt 4:从竞品 / 参考产品反推 PRD

```
我想做一个类似 [竞品名] 的 [功能名]。

帮我:
1. 分析 [竞品] 这个功能的核心逻辑(从用户视角)
2. 它的"隐藏复杂度"在哪(看起来简单但实际难的)
3. 我们的差异化应该在哪(不要完全抄)
4. 基于以上,帮我写一份 PRD 初稿(用 Prompt 1 的结构)

注意:
- 不要抄竞品文案
- 标出"我们可以简化的"和"我们必须做的"
```

### Prompt 5:PRD 改编成"工程师友好版"

```
这是我的 PRD(偏产品视角):[贴]

帮我改编成"工程师友好版":
- 去掉商业背景(工程师不需要)
- 加上技术约束(API 格式 / 数据模型 / 错误码)
- 验收标准改成"可直接写成 test case"的格式
- 加一个"技术方案建议"节(你觉得怎么实现最合理)
- 标出"需要工程师确认"的地方

输出:可以直接贴到 Jira / Linear / GitHub Issue 的格式。
```

## 真实使用 tips

### 1. "非目标"是 PRD 最被低估的节

90% 的 PRD 没写"非目标"。结果:
- 工程师自己"猜"范围
- 做了 PM 没想到的东西
- 上线后"这不是我要的"

**每份 PRD 必须有 3-5 条"非目标"**:
- "本次不做 X"
- "不支持 Y 场景"
- "不考虑 Z 平台"

### 2. 验收标准 = 测试用例的"人话版"

好的验收标准:
> WHEN 用户未登录 THEN 点击"收藏"按钮 SHALL 跳转到登录页,登录后自动收藏

坏的验收标准:
> 用户可以收藏内容

**好的验收标准 = 工程师直接能写 test case**。

### 3. PRD 长度 < 2000 字

超过 2000 字的 PRD **没人看完**。
- 核心需求 < 2000 字
- 详细设计(数据模型 / API)放附录或单独文档
- 图 > 文字(一张流程图 = 500 字)

### 4. 用 Mermaid 画流程图

PRD 里加 mermaid:

```
> 帮我画这个功能的用户流程图(mermaid):
> - 从用户进入页面开始
> - 到完成操作结束
> - 标出所有"分叉"(条件判断)
> - 标出错误路径
```

**一张图让工程师 30 秒看懂整个流程**。

### 5. 写完让 AI 模拟"3 种人"review

```
分别扮演:
1. 资深工程师(关注可行性 / 边界)
2. QA(关注测试覆盖 / 边界条件)
3. 设计师(关注用户体验 / 交互细节)

各自对这份 PRD 提 3 个问题。
```

**一次 prompt 覆盖 3 个视角**,比你自己想全面。

### 6. PRD 不是"写完就完"

PRD 是**活文档**:
- 开发过程中发现问题 → 更新 PRD
- 上线后数据反馈 → 更新 PRD
- 下个迭代 → 基于 PRD 改

**用 Notion / 飞书 + 版本历史**,不要用 Word 邮件来回。

## 真实案例

我自己写 [content-os 的 xiaohongshu-publisher](https://github.com/survivorff/content-os/tree/main/skills/xiaohongshu-publisher) 的 PRD:

- 用 Kiro Spec 模式
- 一句话需求:"半自动发布小红书,Lark 提醒,等用户确认"
- AI 扩成 requirements(15 分钟)
- 我 review + 改(10 分钟)
- AI 出 design(10 分钟)
- 我 review + 改(5 分钟)
- **总 40 分钟出完整 PRD + 设计 + 任务列表**
- 手做估计 4 小时

---

## 这套方案的边界

⚠️ 什么时候不适用:

- **极创新性产品**(没有参考 / 没有竞品):AI 只能从训练数据里找近似,**真创新要你自己想**
- **极复杂业务规则**(金融 / 医疗 / 法律):AI 不懂你行业的隐含规则,**必须人补**
- **政治 / 组织博弈**:PRD 有时候是"说服老板"的工具,AI 不懂你公司的人际
- **极短需求**(改一个按钮颜色):不需要 PRD,直接 Jira ticket
