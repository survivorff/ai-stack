# 🏗 API / 架构设计

> 写代码之前最值得花时间的事。**AI 在这件事上的价值比"帮你写代码"大得多** —— 它能让你把一个错的设计在动笔前就否掉。

---

## 你为什么读到这里?(痛点)

- **没经验**:第一次设计某种系统(IM / 推送 / 支付 / 多租户),不知道有什么成熟模式
- **命名 / 协议 / 约定不一致**:团队 5 个 endpoint,5 种命名风格
- **不知道用什么模式**:REST 还是 GraphQL 还是 gRPC?同步还是事件驱动?
- **性能 / 扩展没想清楚**:写完才发现 N+1 / 单点 / 状态膨胀
- **设计文档没人看**:写了一份长文档,review 的人扫一眼没看懂

---

## 推荐方案

### 🥇 方案 A:用 Kiro Spec 模式(推荐)

**适合**: 中大型功能 / 新模块从 0 开始
**时间**: 2-4 小时
**工具**: [Kiro](../tools/ide/kiro.md) + Claude Sonnet 4.7
**预算**: $19/月

**Kiro 的 Spec 模式天生为这个场景设计**:
1. `requirements.md` —— AI 帮你扩展需求(EARS 格式)
2. `design.md` —— AI + 你做架构 / API / 数据模型
3. `tasks.md` —— AI 拆成可执行任务

**这个流程逼你想清楚再动**。

### 🥈 方案 B:Claude / GPT 网页对话

**适合**: 想清楚思路,不需要立即落 IDE
**时间**: 30 分钟 - 1 小时
**工具**: [claude.ai](../tools/foundation-models/claude.md) / [chatgpt.com](../tools/foundation-models/chatgpt.md)

**步骤**:
1. 描述背景 + 需求 + 约束
2. 让 AI 给 2-3 个方案 + trade-off
3. 反复追问推敲
4. 最后让它写一份"design.md"

### 🥉 方案 C:画图驱动(Mermaid + AI)

**适合**: 视觉学习者 / 给团队 review
**工具**: Cursor / Claude Code + Mermaid

```
> 给我一份这个系统的:
> 1. 架构图(组件 + 数据流)
> 2. 时序图(关键请求路径)
> 3. 数据模型 ERD
> 4. 关键状态机(如果有)
>
> 用 mermaid 写,我直接渲染。
```

## 关键 Prompt 模板

### Prompt 1:让 AI 帮你想全(发散)

```
我要设计一个 [系统/API/模块] 解决 [问题]。
背景:[业务背景 + 现有系统]
约束:[技术 / 性能 / 时间 / 团队规模]
预期规模:[QPS / 数据量 / 用户量]

不要给方案。先帮我想:

1. 这个问题的"本质"是什么?有哪些近似的成熟系统可以参考?
2. 至少 3 种不同的设计方向(架构层面,不是细节)
3. 每个方向的 trade-off:简单 vs 灵活 / 性能 vs 一致性 / 现在 vs 未来
4. 我应该问自己什么问题来选方向

我看完后告诉你选哪个方向,再深入。
```

**为什么这样**:**80% 设计错误是因为"一上来就选了一个方向"**。这个 prompt 强制发散。

### Prompt 2:对一个方案做"压力测试"

```
我倾向用方案 X 来解决 [问题]。
方案描述:[详细描述]

帮我"压力测试"这个方案:

1. 找出 5 个最可能让方案崩溃的边界场景
2. 对每个场景,描述:
   - 何时发生
   - 现有方案怎么失败
   - 修复需要改什么
3. 最危险的场景是哪个?
4. 哪些场景可以"忽略不管"(可接受的边界)?

我宁可现在被骂,也不想上线后被打脸。
```

### Prompt 3:对比 2 个方案做决策

```
我在 X 和 Y 之间选不了:
方案 X:[描述]
方案 Y:[描述]

帮我做决策矩阵:

| 维度 | X | Y |
|------|---|---|
| 实现复杂度 | ? | ? |
| 性能上限 | ? | ? |
| 扩展性 | ? | ? |
| 学习成本(团队) | ? | ? |
| 维护成本(2 年后) | ? | ? |
| 失败时的恢复成本 | ? | ? |
| 我们的痛点匹配度 | ? | ? |

每格简短说明。最后给推荐 + 理由。
```

### Prompt 4:写 design.md

```
基于刚才的讨论,帮我写一份 design.md:

结构:
1. 背景 / 问题(为什么要做)
2. 目标 / 非目标(什么算成功 / 什么不在这次范围)
3. 高层方案(一段话 + 一张图)
4. 详细设计:
   - 组件拆分
   - 数据模型
   - 关键 API
   - 关键时序
5. 替代方案(为什么不选)
6. 风险 + 应对
7. 后续 milestone

不要太长(< 1500 字)。每节简洁,关键内容用 mermaid / 表格。
```

### Prompt 5:RESTful API 命名一致性

```
这是我准备做的 API 列表:[贴 endpoints]

帮我:
1. 找出命名不一致的地方
2. 找出 HTTP method 用错的地方
3. 找出 URL 结构不规范的(嵌套 / 单复数 / 大小写)
4. 重新统一,给出"修正后的列表"
5. 写一份 1 页的"团队 API 命名约定"

不要不必要地引入新规则,基于行业标准 + 我们已有的多数。
```

## 真实使用 tips

### 1. 设计阶段优先用 Opus / GPT-5 Pro

短任务用 Sonnet / GPT-5 标准,**设计阶段用 Opus / Pro**:
- 复杂推理质量差距明显
- 设计错一次代价 5-50 倍写代码错
- 多花一点 token 钱在设计阶段是最划算的

### 2. 逼 AI 先发散再收敛

90% 的人 prompt 习惯:
> "我要做 X,给我设计"

这样 AI 会**直接给一个方案**(且通常是它"觉得最常见的")。
正确姿势:
> "**先列 3 个方向 + trade-off,我选了再深入**"

强制 AI 发散 → 你看到对比 → 决策更明智。

### 3. 用"假装我是面试官"角色让 AI 挑刺

设计完了让 AI 反过来挑刺:
```
> 你现在是 staff engineer 面试官,
> 我刚跟你描述了我的系统设计。
> 你要从专业角度问我 5 个最难答的问题,
> 让我意识到设计里没想清楚的地方。
```

**这一步能挑出"自己想不到的盲区"**。

### 4. 设计文档要有"非目标"

design.md 里**写明"哪些不做"** 比"做什么"更重要:
- 减少范围争议
- 防止 reviewer 误判
- 给后续扩展留空间

让 AI 写时**强调 "非目标 / out-of-scope" 这一节**。

### 5. 不要让 AI 替你拍板技术选型

AI 会给"行业最常见"的答案,**它不知道你团队的具体情况**:
- 团队没人会的技术 → 哪怕 AI 说"是最佳" 也别选
- 公司其他系统都用 X → AI 推荐 Y 也别动(一致性 > 局部最优)
- 历史包袱重 → AI 不知道,只看技术维度

**AI 给方案,你做决策**。

### 6. 设计文档要给"决策日志"

```
ADR-001:为什么选 PostgreSQL 不选 MongoDB?
ADR-002:为什么用事件驱动而不是同步 RPC?
ADR-003:为什么 cache 用 Redis 不用本地?
```

让 AI 帮你把每个关键决策写成 ADR 格式。
**3 年后接手项目的人会感谢你**。

## 真实案例

我设计 [content-os 的 xiaohongshu-publisher 工作流](https://github.com/survivorff/content-os/tree/main/skills/xiaohongshu-publisher):

- Kiro Spec 写 requirements(20 分钟):理清"半自动 / Lark 提醒 / 用户确认"
- design.md(40 分钟):把流程拆成 5 个阶段
- 跟 Claude 反复推敲"用户确认逻辑怎么设计最少打扰"
- 最终落地比"yolo 直接写代码" 省 60% 后续修改时间

---

## 这套方案的边界

- **真正创新性的设计**(全新概念 / 没先例):AI 只能从训练数据里找近似,**真创新还是要靠人**
- **公司政治 / 跨团队协调**:AI 不懂你团队的人际 / 利益结构
- **极致优化**(纳秒级延迟 / 内存极限):AI 给不了硬件级优化
- **一些行业 know-how**(金融 / 医疗 / 安全合规)需要人补
