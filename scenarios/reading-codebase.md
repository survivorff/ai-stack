# 📖 读懂老项目 / 陌生代码库

> "新人入职 → 接手一个老项目"是工程师最折磨的时刻之一。AI 工具能把这个过程**从一周缩短到一天**。

---

## 你为什么读到这里?(痛点)

- **不知道从哪开始**:克隆下来 100+ 文件,目录看着像迷宫
- **没文档 / 文档过期**:README 还是 3 年前写的
- **跑都跑不起来**:依赖装不上 / 配置缺失 / 数据要怎么弄
- **业务逻辑藏在屎山**:某个核心模块 2000 行,过去 5 个工程师轮番改过
- **决策原因在 git history**:为什么这么写?2 年前某个 commit 的 message 写"hotfix"

---

## 推荐方案

### 🥇 方案 A:Claude Code 整体扫描(主力)

**适合**: 任何陌生项目接手
**时间**: 30 分钟 - 2 小时
**工具**: [Claude Code](../tools/coding-agent/claude-code.md)
**预算**: Claude Pro $20+

**步骤**:
1. `cd` 到项目目录
2. 启动 `claude`
3. 跑下面 4 个 prompt(下方有模板)
4. 拿到一份完整的"项目地图"

### 🥈 方案 B:Kimi 长上下文塞整项目

**适合**: 中小项目(< 100 文件)
**工具**: [Kimi K2.5](../tools/foundation-models/kimi.md) Web
**预算**: ¥0(免费)

**步骤**:
1. 项目代码打成 zip 上传 Kimi
2. 让 Kimi 做"完整项目分析"
3. 边对话边追问

**Kimi 2M 长上下文 + 国内直连**:接手陌生项目首选。

### 🥉 方案 C:Cursor 边读边问

**适合**: 边读代码边随时问 AI
**工具**: [Cursor](../tools/ide/cursor.md)

进 Cursor 打开项目 → `@codebase` 让它索引 → 边读边 Cmd+L 问。

## 关键 Prompt 模板

### Prompt 1:项目第一印象(进项目跑的第一个 prompt)

```
我刚接手这个项目。@codebase

帮我做一份"30 秒入门简介",回答:
1. 这是什么项目?(一句话定位)
2. 用什么技术栈?(语言 + 框架 + DB + 部署)
3. 项目的入口在哪?(从哪开始读)
4. 核心业务模块是哪几个?(列 3-5 个,各是什么职责)
5. 哪些目录可以"先不看"?(测试 / 配置 / 第三方 / build)

不要逐个文件解释,只给"全局印象"。
```

### Prompt 2:画项目架构图

```
基于刚才的认知,画 mermaid 图:
1. 整体架构图(模块之间的依赖关系)
2. 一个核心请求的时序图(从入口到落库)
3. 数据模型图(主要的实体和关系)

每个图都要简化,不要把所有细节都画上,
只保留"理解项目必须看懂"的那部分。
```

**Mermaid 图直接渲染**(在 Cursor / VS Code 里 / Claude Code 也行)。

### Prompt 3:让 AI 找"难懂的地方"

```
这个项目里,有哪 5-10 个地方:
- 代码看起来不像"标准写法"(可能有历史包袱)
- 有 TODO / FIXME / HACK 注释
- 函数特别长 / 嵌套特别深 / 名字特别隐晦
- 多个文件在一起做一件事但没文档解释为什么这么拆

把它们标出来,我就知道哪些地方要小心 + 哪些地方将来要重构。
```

### Prompt 4:写 GETTING_STARTED.md

```
基于你对项目的理解,帮我写一份 GETTING_STARTED.md:

包含:
1. 项目是什么(2-3 句话)
2. 本地怎么跑(步骤 + 命令,实际可行的)
3. 核心目录结构(哪个目录干什么)
4. 我应该先读哪 5 个文件
5. 最常见的"踩坑"和解法

这份文档让"明天来的下一个人"能 1 小时上手项目。
```

**好处**:这份文档**留给项目永久财富**(下个新人不用再问 AI)。

## 真实使用 tips

### 1. 项目跑不起来?直接让 AI 帮你

```
> 我克隆下来跑不起来。报错:[贴报错]
> 帮我写一份"从 0 开始本地启动"的 step-by-step 脚本,
> 包括装依赖、配 env、起 DB、起服务、跑测试。
> 用 macOS 默认环境,假设我已有 Homebrew + Node。
```

**节省 80% "翻文档 + 试错" 的时间**。

### 2. 看不懂某段代码

选中代码 → Cmd+L:
```
> 这段代码在做什么?用我能听懂的话(我是新人)解释。
> 然后告诉我:
> 1. 它被谁调用?
> 2. 它调了什么?
> 3. 如果改坏了,会影响哪些功能?
```

### 3. git blame 让 AI 帮你"考古"

```
> 这个函数 src/legacy/calc.ts 写得很丑。
> 帮我跑 git log 看它的历史:
> 1. 谁写的初版?
> 2. 改过几次?最大的改动是什么时候 / 为什么(看 commit message)?
> 3. 是不是某次 hotfix 之后没重构?
```

**很多屎山是有故事的**。看完 git history 你就理解为什么"它这么丑"。

### 4. 用"提问法"读代码

不要从 file_1.ts 开始一行行读。
带着具体问题去读:
- "这个项目怎么处理用户认证?"
- "下单流程涉及哪些 DB 操作?"
- "怎么部署到生产环境?"

每个问题让 AI 帮你找相关代码 → 你只读相关那部分。

### 5. 输出要落到文档,不只是"看完就忘"

读项目过程中,**让 AI 帮你边读边写**:
- `docs/onboarding-notes.md`(你自己的笔记)
- `docs/architecture.md`(架构图 + 说明)
- `docs/key-modules/*.md`(核心模块 deep dive)

这些文档**比你"读了忘"价值大 10 倍**:
- 你下次不用重新读一遍
- 下个新人能用
- 项目里其他人也会感谢你

### 6. 做新人的"先用 AI 上手再问同事"

很多新人**第一周问同事问 50 次**。同事忙,新人也焦虑。
**先 AI 自己消化 + 列出"剩下没搞清的 3 个问题"** → 再问同事。
- 你的问题更具体更有价值
- 同事感觉你不是"伸手党"
- 你成长更快

## 真实案例

我接手 [hyperliquid_design](https://github.com/survivorff/hyperliquid_design) 之前不熟悉它的源码:

- Kimi 上传项目 zip(免费,2M 上下文够)
- 跑 4 个上面的 prompt(15 分钟)
- 拿到完整架构图 + 核心模块 deep dive(导出 markdown)
- **第一天就能写文章不用慢慢摸**

**总耗时 1.5 小时 vs 手动估计 8-16 小时**。

---

## 这套方案的边界

- **极复杂业务逻辑** AI 仍可能漏掉(产品规则需要人解释)
- **隐式约定** AI 看不出(团队某种命名 / 某种流程惯例)
- **过期 / 错的注释** AI 会信(注释说做 X,代码做 Y,AI 默认信注释)
- **Build / Deploy 系统** 复杂时 AI 给的 quick start 可能不够全
